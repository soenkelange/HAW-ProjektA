=== Projektbeschreibung - Brettspiel Attika, v_1.0.1, 2016-10-10 ===
==============================================================================

Ziel des Projektes soll die Umsetzung des Brettspiels Attika als Webanwendung sein, bei dem zwei bis vier Spieler einen antiken Stadtstaat aufbauen oder zwei sogenannte Heiligtümer miteinander verbinden müssen.

// Startseite
- Zwei bis vier Spieler können über vier Texteingabefelder ihren Namen eingeben.
- Durch eine Schaltfläche können die Spieler ihre Eingabe bestätigen.
- Wenn nicht mindestens zwei Textfelder mit Text befüllt sind, wird den Spielern eine Fehlermeldung angezeigt, sobald das Formular abgeschickt wird
- Wenn mindestens zwei Textfelder mit Text befüllt sind, werden die Spieler zum Spiel weitergeleitet, sobald das Formular abgeschickt wird

// Spielstart
- Abhängig von der Anzahl der eingegebenen Namen wird das Spielfeld generiert
- Des Weiteren wird bei Spielstart der Spieler der Anfängt durch einen Zufallsgenerator ausgelost
- Hinzu kommt, dass alle Spieler jeweils ein Hauptgebäude und drei Nebengebaude durch einen Zufallsgenerator zugelost bekommen
- Abhängig von ihrer Position in der Spielreihenfolge erhalten die Spieler außerdem Resourcen zugelost
- Nach der Initialisierung des Spiels beginnt der erste Zug der Startspielers

// Spielzug
- Zu Beginn seines Spielzuges bekommt der Spieler zwei Schaltflächen angezeigt, über die er sich zum einen für das Aufdecken von Gebäuden oder dem Bauen von Gebäuden entscheiden kann

//// Aufdecken von Gebäuden
- Wenn sich der Spieler für das Aufdecken von Gebäuden entscheidet, in dem er auf die entsprechende Schaltfläche drückt, bekommt der Spieler drei Schaltflächen angezeigt, über die er sich für das Aufdecken eines Hauptgebäudes, eines Nebengebäudes oder dem ziehen von drei Resourcen entscheiden kann
- Wenn sich der Spieler für das Aufdecken eines Haupt- oder Nebengebäudes entscheidet, in dem er auf die entsprechende Schaltfläche drückt, bekommt er durch die Anwendung ein entsprechendes Gebäude per Zufallsgenerator zugelost
- Nachdem dem Spieler durch die Anwendung ein Gebäude zugelost bekommen hat, wird ihm das Gebäude und erneut die vorherige Auswahlmöglichkeit angezeigt, allerdings mit einer reduzierten Anzahl an Resourcen angezeigt.
- Der Spieler kann sich ingesamt drei Mal für das Aufdecken von Gebäuden entscheiden, danach ist sein Spielzug beendet.
- Wenn sich der Spieler für das Aufdecken von Resourcen entscheidet, in dem er auf die entsprechende Schaltfläche drückt, bekommt er durch die Anwendung Resourcen per Zufallsgenerator zugelost und sein Spielzug ist beendet.

//// Bauen von Gebäuden
- Wenn sich der Spieler zu Anfang seines Zuges für das Bauen von Gebäuden entscheided, in dem er auf die entsprechende Schaltfläche drückt, bekommt der Spieler zwei Schaltflächen angezeigt, um sich für das Bauen eines Gebäudes oder dem ziehen von drei Resourcen angezeigt.
- Wenn sich der Spieler für das Bauen eines Gebäudes entscheidet, wird ihm seine Ablagetafel und das Spielfeld angezeigt.
- Auf der Ablagetafel werden die Gebäude dargestellt, die er in vorherigen Zügen gezogen hat.
- Sobald die Ablagetafel angezeigt wird, kann der Spieler eines der auf der Ablagetafel angezeigten Gebäuden auswählen.
- Nachdem der Spieler ein Gebäude ausgewählt hat, wird er aufgefordert einen Bauplatz auf dem Spielfeld auszuwählen.
- Wenn der Spieler einen Bauplatz ausgewählt hat, werden ihm die mit dem Bauplatz und dem Gebäude verbunden Baukosten angezeigt und er kann das Bauen bestätigen oder ablehnen.
- Wenn der Spieler eine Straße als Gebäude ausgewählt hat, dann wird ihm zudem eine Auswahlmöglichkeit angezeigt, um billige Resourcen zum Bauen auszuwählen, sofern dies nötig ist
- Wenn der Spieler das Bauen ablehnt, kann er einen neuen Bauplatz auswählen.
- Wenn der Spieler das Bauen bestätigt, wird das Gebäude auf dem entsprechenden Bauplatz angezeigt. Außerdem wird das Gebäude und die verwendeten Resourcen nicht mehr auf der Ablagetafel angezeigt.
- Sobald der Spieler ein Gebäude gebaut hat, wird ihm erneut die Auswahlmöglichkeit zwischen dem Bauen eines Gebäudes und dem ziehen von nun mehr zwei Resourcen angezeigt.
- Der Spieler kann sich ingesamt drei Mal für das Bauen von Gebäuden entscheiden, danach ist sein Spielzug beendet.
- Wenn sich der Spieler für das Aufdecken von Resourcen entscheidet, in dem er auf die entsprechende Schaltfläche drückt, bekommt er durch die Anwendung Resourcen per Zufallsgenerator zugelost und sein Spielzug ist beendet.

//// Spielzugende
- Bevor der Spielzug eines Spielers entgültig beendet ist, wird geprüft, ob der Spieler über sog. Amphoren verfügt.
- Wenn der Spieler über Amphoren verfügt, bekommt der Spieler einen Hinweis, dass er seinen Spielzug verlängern kann, und zwei Schaltflächen zum Verlängern des Spielzugs bzw. zum nicht Verlängern angezeigt.
- Entscheidet sich der Spieler für das Verlängern seines Zuges, dass war je nachdem was er zuvor gewählt hat, noch ein Gebäude bauen, ein Gebäude aufdecken oder er bekommt noch eine Resource zugelost.
- Sobald der Spieler alle seine Möglichkeiten ausgespielt hat, wird geprüft ob der Spieler eine Amphore durch zusammenhängendes Bauen von Gruppen verdient hat
- Wenn der Spieler eine Amphore verdient hat, wird ihm dies für einen bestimmten Zeitraum angezeigt.
- Anschließend wird er weitergeleitet und ihm wird angezeigt, welcher Spieler nun an der Reihe ist.

# Planning Poker
Projektidee ist es eine Anwendung zu erstellen mit der Teams ein Planning Poker,
auch Scrum Poker genannt, durchführen können.

Product Owner können in einem gesichterten Bereich der Anwendung ein Product
Backlog erstellen. In diesem Bereich ist es möglich neue User Stories mit Titel
und Beschreibung zu erstellen oder zu importieren. Des Weiteren können bereits
vorhandene User Stories bearbeitet oder gelöscht werden. Außerdem können die
User Stories innerhalb des Product Backlogs priorisiert werden.
In dem gesicherten Bereich kann ebenfalls ein Link generiert werden, über den
andere Teammitglieder an der Planning Poker Sitzung teilnehmen können.

Um an einer Planning Poker Sitzung teilnehmen zu können, müssen die
Teammitglieder den generierten Link öffnen und zunächst ihren Namen eingeben.

Anschließend wird die erste User Story dem Teammitglied präsentiert. Das
Teammitglied hat dann die Möglichkeit seine Aufwandsschätzung auszuwählen.
Nachdem alle Teammitglieder eine Aufwandsschätzung ausgewählt haben, wird allen
Teammitgliedern das Ergebnis angezeigt. Der Product Owner hat dann die
Möglichkeit, der User Story eine feste Aufwandsschätzung zuzuordnen.

Nachdem der User Story eine feste Aufwandsschätzung hinzugefügt wurde, wird
allen Teammitgliedern die nächste User Story angezeigt und der Ablauf wiederholt
sich.

Die Sitzung ist beendet, sobald alle Teammitglieder die Sitzung verlassen haben,
oder keine weiteren ungeschätzten User Stories vorhanden sind.

Hmm... Sie haben hier beschrieben, was Planning Poker (eine Methode der agilen Softwareentwicklung) ist, bzw. wie es abläuft. Der zentrale Dreh- und Angelpunkt von Planning Poker ist die Interaktion zwischen Menschen, die sich (ob nun in einem Raum oder über eine Konferenzschaltung) gegenüber stehen oder sitzen. Das bedeutet, dass die eigentliche Interaktion kein Teil Ihres Programms ist. Damit bleibt für die Anwendungen nur noch die Speicherung der Projektvorschläge und die Bewertung davon.

Leider beschreibt Ihr Vorschlag damit keine gute Projektidee im Sinne einer interaktiven Anwendung, weil es sich um eine Datenbankanwendung handelt. Das bedeutet, dass es bei der Umsetzung Ihres Projekts vorrangig darum geht, das NutzerInnen Texte (und ggf. Dateien) hochladen können und sich diese anzeigen lassen können.

Außerdem sollte es etwas sein, dass es noch nicht gibt. Nun bin ich mir nicht sicher, ob es die entsprechende Anwendung zu Planning Poker schon gibt, aber es würde mich doch sehr wundern, wenn nicht. (Aber da müsste ich einen Scrum-Master fragen, der dürfte es wissen.)

Was Sie dagegen ausarbeiten sollten war eine Projektidee mit verschiedenen Interaktionsmöglichkeiten. Das bedeutet, dass es etwas sein soll, bei dem NutzerInnen Eingaben machen, Schaltflächen anwählen usw. usf. und das Programm reagiert dann darauf in abwechselungsreicher und interessanter Art und Weise. Spiele sind hierfür die anschaulichsten Beispiele.

Bitte erstellen Sie deshalb eine Projektbeschreibung, bei der Sie das folgende Spiel umsetzen: Attika http://www.brettspiele-report.de/attika/
